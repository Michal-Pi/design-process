# W3C Design Tokens Community Group Format — v2025.10

<!-- complete-design reference: dtcg-v2025-10 -->

## Purpose

DTCG v2025.10 is the first stable W3C spec for design token interchange. Design-os uses it
as the canonical format for Stage 5b token emission. Pin to v2025.10; breaking changes
require a major schema version bump in `schemas/src/`.

## Citations

- W3C DTCG (2025-10-28). *Design Tokens Format Module*, v2025.10.
  https://www.designtokens.org/tr/2025.10/format/
- §1.1: "A design token is a specific value to be applied for a specific use case in a
  specific design context." The `$type` field carries the use-case declaration.
- §3.1 Token file format: top-level group key → `$value`, `$type`, `$description`, `$extensions`.
  Design-os uses `$extensions.complete-design` for structured provenance metadata.
- §4.1 Primitive tier: raw named values (`--color-blue-500`). Stage 5b emits primitives first.
- §4.2 Semantic tier: role-based aliases (`--color-primary`). Stage 5b maps primitives to roles.
- §4.3 Component tier: component-scoped overrides (`--button-background`). Stage 5b emits
  component tier only for components with ≥3 token overrides.
- §5.1 media type: `application/design-tokens+json`; file extension `.tokens.json`.
  `schemas/dist/` uses `.tokens.json` for all DTCG output.
- §6.1: `$value` aliasing syntax `{color.primary.500}` for semantic references.

## How complete-design uses it

- Stage 5b: `dtcg-lint.mjs` validates every emitted `.tokens.json` against v2025.10.
- `schemas/src/` DTCG Zod schema tracks §3.1 + §4.x tiers.
- DESIGN.md frontmatter token fields map to DTCG semantic tier.
