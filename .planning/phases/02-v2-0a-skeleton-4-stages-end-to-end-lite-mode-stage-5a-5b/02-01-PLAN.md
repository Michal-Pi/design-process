## Plan 02-01 — Stage 1 Gate Business Logic + Discover Workflow SKILL.md + Adversarial CI (RED-01..06)

**Goal:** Fulfill ROADMAP Phase 2 Success Criterion 2: "The Stage 1 gate hard-blocks `evidence: VALIDATED` 100/100 times when fed only synthetic personas (RED-05 adversarial CI); a prompt-injection canary asserts the red line cannot be bypassed by creative prompting (RED-06); every `findings.md` propagates `worstProvenance:` from cited personas." This plan also ships the `discover` workflow SKILL.md and its three atoms, which are required for Plans 02-02..02-05 to have a runnable Stage 1 to precede them.

**Locked decisions consumed:** D-32 (structured-frontmatter SKILL.md), D-33 (atom dual-path standalone-bootstrap), D-34 (one sub-agent per stage), D-36 (personas-proto: Indi Young format + provenance:generated + ASSUMPTIONS.md), D-37 (synthetic-persona hard-block is a script, not LLM), D-38 (worstProvenance propagation via frontmatter-validate.mjs extension), D-49 (cost gates: discover p50 ≤30k, hard-stop 60k), D-50 (three adversarial CI suites), D-51 (INFERRED labeling discipline), D-53 (host fallback section in each SKILL.md)

**Phase 1 dependencies:** `assets/scripts/gates/base.mjs` (runGate base + 5-kind GateResult), `assets/scripts/gates/stage-1.mjs` (skeleton to fill in), `assets/scripts/frontmatter-validate.mjs` (extend with --check-worst-provenance), `assets/scripts/handoff-bundle-build.mjs` (emit stage-1 bundle), `assets/scripts/manifest-lock-append.mjs` (append gate result), `references/indi-young-thinking-styles.md` (persona prompt template), `references/torres-ost.md` (OST format reference), `references/klement-jtbd.md` (JTBD format), `schemas/dist/persona.v1.json` (schema validation), `evals/bundles/fixtures/stage-1-to-2/` (placeholder bundle to replace)

**Open research flags resolved in this plan:**
- OF-02: `worstProvenance` field placement — top-level YAML frontmatter on every downstream artifact (same level as `artifact:`, `stage:`, `generated:`). This is enforced by extending `frontmatter-validate.mjs` with a `--check-worst-provenance` mode that reads the frontmatter of `findings.md`, `sitemap.json`, and any artifact whose frontmatter contains a `cites:` list of persona file paths.

---

### Anti-patterns to avoid

- **Pitfall 2 (synthetic-persona red line breach):** Do NOT write gate logic that can be bypassed by a `provenance` field nested inside a JSON body field rather than in the YAML frontmatter. The gate reads YAML frontmatter of `.persona.json` files (the `provenance:` field is in YAML-style frontmatter at the top of each JSON file using gray-matter). Personas without frontmatter are treated as `provenance: missing` (most conservative).
- **Anti-Pattern 3 (LLM as data validator):** Gate logic must be pure script — no LLM calls in `stage-1.mjs`. The lint-determinism CI gate enforces this already.
- **OF-05 resolved:** RED-05 adversarial tests are pure script tests — each fixture is a pre-built `design/` directory with synthetic-only personas already written (no LLM call needed). The gate is invoked directly via `runGate('1', designDir, {})`.
